# How to Train a Finance Deep Research Agent: Design Principles and Lessons Learned

我们从零构建并用强化学习训练了一个金融深度研究 Agent。本文分享核心设计理念、关键决策，以及落地过程中的经验教训。代码与配置已在 AgentJet 全面开源。

## 1 引言

### 1.1 定义金融 Deep Research

想象你是金融分析师，收到这样的需求："分析近两年光伏行业的产能过剩情况，并评估其对头部企业盈利能力的影响。"

这类问题的难点不只是"写出来"，而是"研究清楚"。真实投研要先形成初步判断，理清主线：看哪些宏观指标，比哪些财报，补哪些资料，预判哪些结论可能被推翻。接下来才是围绕主线展开：查数据、读财报、检索研报、交叉验证，并持续修正假设。

金融 Deep Research Agent 的目标，正是把这套过程自动化：不是生成一篇看起来像样的报告，而是通过多轮规划、检索、验证和修正，产出论据扎实、逻辑严密的研究结果。

### 1.2 核心挑战：多维目标的内在冲突

构建这样一个 Agent，需要在三个相互牵制的维度中寻找平衡：

- **证据可追溯**：金融场景对幻觉零容忍，每个核心数据和结论都必须有来源支撑。
- **分析充分性**：不仅要完成信息检索，还要识别矛盾、形成分析框架，给出有价值的增量判断。
- **可读性与可用性**：产出必须具备高信息密度和清晰结构，能直接作为决策参考。

这三者彼此拉扯。过分强调事实性，模型容易变成"资料搬运工"；过分鼓励分析深度，又容易滑向站不住脚的"精彩幻觉"；过于偏好规整结构，模型可能减少必要的探索和验证。

### 1.3 现有范式的局限

**Workflow 的问题在于流程太固定。** 预定义的 Multi-Agent 编排适合边界清晰的任务，但难以应对真实投研中不断变化的问题空间。

**SFT 的问题在于它更容易学到语言风格，而非分析能力。** 模型可以学会专业风格和篇章结构，却难以学到隐含的分析路径：如何拆解问题、处理矛盾、回头查证。

### 1.4 我们的方案

当意识到 SFT 只能决定能力下限时，转向强化学习就成了必然选择。我们的核心思路是从"隐式行为模仿"到"显式目标优化"：明确定义什么是高质量金融研究，让模型在真实工具环境中自主探索最佳策略。

要让 RL 飞轮真正转动，必须跨越三道屏障：

1. **衡量"研究质量"的可靠标尺**：从零构建形式化评估基准
2. **突破训练数据的成本瓶颈**：将需求从"标准答案"降维到"高质量设问"
3. **构建全链路动态训练环境**：设计防 Reward Hacking 的多维体系，搭建高并发工具缓存架构

## 2 评测标准设计

### 2.1 现有 Benchmark 的局限性

现有金融 benchmark（如 FinBen[1]）大多面向单轮、封闭式任务，适合评估知识提取，而非研究执行。Deep Research Agent 的关键能力在于：能否围绕未知问题主动规划工具调用、能否在长上下文中组织证据并保持逻辑一致、能否完成开放式分析。

### 2.2 我们的 Benchmark

我们的 benchmark 覆盖五类任务：**宏观分析、行业研究、事件解读、个股分析和公司研究**，每个 domain 6 个代表性 query，共 30 个样本。题目由资深投研专业人士设计，并提供参考答案。

按 domain 拆分比只看综合分更有诊断价值——模型可能擅长宏观叙事，却在财务数据对齐上失误。相比扩大样本量，我们更看重 query 的代表性和区分度。

### 2.3 评测重点：分析充分性

我们最关注**分析充分性**：模型是否真正组织了证据，是否完成了有支撑的推导。我们采用 pairwise 评估，将模型输出与专家参考报告对比。相对比较通常比绝对评分更稳定，参考报告提供基线而非唯一标准答案。

### 2.4 评测与训练目标的差别

**适合做 evaluation 的指标，不一定适合直接做 RL reward。** 如果只看"分析是否充分"而不约束数据真实性，模型就可能跳过工具调用，直接编造数据。这是为什么在 benchmark 之外，我们还需要单独设计更完整的 reward 体系。

## 3 训练方案设计

### 3.1 训练数据：从标准答案转向高质量问题

RL 不依赖大量 `(question, gold answer)` 对，而依赖高质量 query 和合理 reward。训练集覆盖 5 个 domain，与评测集严格隔离。

我们设计了专用于金融的问题合成策略：Agent 先按混合比例选择研究方向，进行头脑风暴，结合真实工具收集信息，得到有挑战性的金融问题并以多种口吻表述。最终收集约一千条高质量训练 Query。

### 3.2 奖励设计：多维 Reward

最初尝试把"分析充分性"直接作为 reward，但模型容易走捷径：编造数据或写出形式完整但分析不足的报告。

#### 核心目标与约束项

我们把 reward 拆成 **1 个核心目标 + 3 个约束项**：
- **核心目标**：分析充分性（rm reward）
- **约束项**：事实性（audit）、引用规范（grounding）、呈现质量（presentation）

![img](https://img.alicdn.com/imgextra/i4/O1CN01KOkk6021pd3j6ysIa_!!6000000007034-55-tps-927-20.svg)

分析充分性是金融 Deep Research 最核心的能力；事实性、引用和呈现质量承担"约束"作用，防止跑偏。

#### 基于规则计分

我们把评分拆成两步：**先抽取，再计分**。LLM 先提取关键信息、引用和证据关系，规则代码再按预定义逻辑计算各维度得分。这样分数更稳定，也更容易排查问题。

#### 负向惩罚

我们加入惩罚项限制无效策略。例如模型在整轮 rollout 中几乎不调用工具，则施加负向惩罚。这在训练初期可先排除明显错误方向。

### 3.3 预定义流程

即使 reward 合理，让模型一开始就完全自由探索效率不高。最常见的问题是"没有形成完整研究过程"：查到信息就立刻开始局部分析，报告结构松散。

我们在训练初期提供简单研究框架：**先规划，再执行**。模型先给出分析提纲，再进入多轮工具交互和写作，形成稳定的研究展开方式。

## 4 训练基建

在强化学习里，工具系统本身就成了训练环境的一部分，直接影响 Reward 和梯度更新。推理阶段能容忍的小问题——偶发超时、接口抖动——到训练阶段都会被大规模并行 rollout 成倍放大。

### 4.1 从"能用的工具"到"可训练的环境"

金融 Deep Research 需要高信噪比的结构化信息。我们把工具系统封装成 [**Finance-MCP**](https://github.com/flowllm-ai/finance-mcp) 并开源，把常用的数据获取、网页抓取和搜索能力收在同一个 MCP 接口下。

我们还把**同花顺网页内容预先抓取和缓存**。很多信息散落在半结构化网页中，实时访问成本高且结果随时间波动。缓存能让训练环境更稳定，减少外部变化对 reward 的干扰。

### 4.2 训练环境的三个问题

**成本**：GRPO 的 group rollout 会重复调用同样的工具。我们把工具执行解耦成 **EnvService**，加 **MongoDB 缓存**，同一个 `(tool_name, arguments)` 只执行一次。

**确定性**：缓存保证同样输入得到同样结果，提高实验可复现性。

**鲁棒性**：原则是**单点失败不能拖垮整条训练链路**。工具调用失败时重试并返回错误信息；结构化解析做容错修复；评分失败返回 `score=0` 并记录日志；judge LLM 配重试和降级策略。

## 5 实验结果

延续前面的设计，最终 reward 由 **1 个核心目标和 3 个约束项** 组成：
**rm** 对应分析充分性，**audit** 对应事实性，**grounding** 对应引用规范，**presentation** 对应呈现质量。

## 6 踩坑与反思

### 6.1 环境不稳定会放大"走捷径"倾向

在真实金融 API 环境中，限流、超时和抖动难以避免。如果直接暴露给模型，策略会倾向于规避真实工具调用，转而编造数据。训练侧的缓存、状态冻结和错误隔离不只是工程优化，而是训练设计的一部分，减少环境噪声对策略学习的干扰。

### 6.2 不同能力的提升通常不同步

多维 reward 下，容易优化的部分先提升，综合能力提升较慢。分析充分性是最终主要收益来源，事实性、引用和写作质量更多表现为波动中逐步稳定。只要整体趋势没有失稳，不需要过早因短期起伏频繁调参。

### 6.3 评测信号的稳定性往往比想象中更重要

在 GRPO 中，奖励信号方差直接影响优势估计质量。我们把"理解"与"打分"拆开：LLM 负责语义判断和结构化抽取，最终分值交给确定性规则计算。一个足够稳定的评测系统，很多时候比更复杂的训练技巧更重要。

### 最小复现关键点

| 核心模块         | 最低要求                                                 |
| ---------------- | -------------------------------------------------------- |
| Foundation Model | 具备基础多轮工具调用能力，或已通过 SFT 对齐到可用起点    |
| Data Engine      | 少量但具备多样性的 seed queries，不依赖标准答案          |
| Tool Execution   | 统一的结构化工具接口，支持调用链路追踪                   |
| Reward Design    | 以分析充分性为核心，同时保留事实性、引用和呈现质量等约束 |
| Evaluation Infra | 语义抽取与规则算分结合的稳定评测流程                     |
| Training Infra   | 能隔离环境噪声、缓存重复调用，并对边缘异常做安全降级     |

**一句话总结**：训练金融 Deep Research Agent，本质上不是教模型生成看起来更完整的报告，而是定义什么是好的研究，并把它转化为可评估、可反馈、可优化的训练信号；当这套机制足够清晰稳定，模型才有可能逐步学会真正有效的研究策略。
